项目开发规范

两类规范

  • 非编码类规范：开源规范、文档规范、版本规范、Commit规范和发布规范
  • 编码类规范：目录规范、代码规范、接口规范、日志规范和错误规范

非编码类规范

开源规范

开源协议

​ 开源项目一定需要一个开源协议，规定了使用开源软件时的权力和责任

​ 常见的开源协议有6种：GPL、MPL、LGPL、Apache、BSD、MIT。

开源规范特点

  • 高的单元测试覆盖率
  • 代码和提交记录种不能出现内部IP、内部域名、密码和密钥等信息，避免造成敏感信息的泄露
  • 开源项目被别的开发正提交pull request、issue和评论时要及时处理
  • 持续更新功能、修复bug

项目文档规范

​ 包含了一切需要文档化的内容，集中放在项目/docs目录下。

​ 不同的项目有不同的文档需求，在指定文档规范时，可以考虑包含两类文档：

  • 开发文档：说明项目的开发流程，包括开发环境搭建的流程、构建二进制文件、测试和部署
  • 用户文档：软件的使用文档，对象是用户。可以包含API文档、SDK文档、安装文档、功能介绍文档、 最佳实践、操作指南、常见问题等。

API建接口文档规范

​ API文档描述了组件提供的API接口介绍及调用方法。在前后端分离的场景可以为前端开发人员提供指导，减少沟通成本，提供开发效率。

编写API文档的方式

​ 接口文档有四种编写方式：

  • Wrd格式文档
  • 借助工具编写
  • 通过注释生成：swagger
  • 编写Markdown文档

​ 使用Markdown格式编写文档的优势：

  • 相比于swagger，markdown可以表达更丰富的内容和格式，且不需要在代码内添加大量的文档
  • 相比于Word格式，markdown文档占用空间更小，能够跟随单马仓库一起发布，便于查找
  • 相比于API文档编写工具，markdown文档免去了怼第三方平台依赖和网络限制

API文档的内容

  • 完整的API接口介绍文档
  • API接口变更历史文档
  • 通用说明
  • 数据结构说明
  • 错误码描述
  • API接口使用文档

本项目API文档组成

  • README.md：API接口介绍文档
  • CHANGELOG.md：变更历史
  • generic.md：用用的请求参数、返回参数、认证方法和请求方法等
  • struct.md：接口文档种所使用的数据结构
  • user.md, sercet.md, polict.md：API接口文档
  • error_code.md：错误码描述

API接口文档内容

  • 接口描述
  • 请求方法url+METHOD
  • 输入参数：参数名称、是否必选、类型、描述
  • 输出参数：参数名称、类型、描述
  • 请求示例

版本规范

使用版本好的意义：

  • 通过版本号，快速定位问题的位置
  • 使得使用者了解项目当前进度

语义化版本规范

​ SemVer, Semantic Versioning是Github起草的具有指导意义的、统一的版本号表示规范，

​ 它规定了版本号的表示、增加和比较方法，以及不同版本号代表的含义。

​ 版本号及其更新方式包含了相邻版之间的底层代码和修改内容的信息。

  • 主版本号(MAJOR)：当作了不兼容的API修改
  • 次版本号(MINOR)：向下兼容的功能性新增和修改。偶数为稳定版本，奇数为开发版本
  • 修订号(PATCH)：做了向下兼容的问题修正

先行版本号

​ 有时候还会有先行版本号：X.Y.Z[-先行版本号] [+版本编译元数据]

  • 先行版本号：意味者该版本不稳定，可能存在兼容性问题

  • 编译版本号：在编译过程种自动生成，不进行认为控制

    举几个例子：

  • 1.0.0-alpha+001

  • 1.0.0+20130313144700

  • 1.0.0-beta+exp.sha.5114f85

语义化版本控制规范

  • 标记版本后发行后，禁止修改该版本软件的内容，任何修改都需要发布新的版本
  • 主版本号为0的软件处于开发初始阶段，一切都可能随时被修改，1.0.0版本为第一个稳定版本，之后所有的版本号更新都基于该版本进行修改
  • 修订号Z必须在只做了向下兼容的修正（进行了原有功能的bug修复）时才递增
  • 次版本号Y必须在向下兼容新功能出现时递增，在任何公共API功能被标记为弃用时，也必须递增，当有改进时，也可以递增。当Y递增时，修订号归零
  • 主版本X号必须在有任何不兼容的修改被加入公共 API 时递增。其中可以包括次版本号及修订级别的改变。每当主版本号递增时，次版本号和修订号必须归零。
